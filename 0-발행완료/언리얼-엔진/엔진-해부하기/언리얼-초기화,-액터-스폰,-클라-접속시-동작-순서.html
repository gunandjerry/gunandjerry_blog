언리얼 초기화, 액터 스폰, 클라 접속시 동작 순서

엔진 초기화 -> 레벨과 함께 로드된(맵 패키지에 포함되어 있던) 액터의 초기화

보다 자세한 내용은 문서 참고.

  1. GEngineLoop.Init()
    1. UGameEngine::Init(IEngineLoop* InEngineLoop)
      • UGameInstance, UGameViewportClient, ULocalPlayer 생성
    2. UGameEngine::Start()
      1. UGameInstance::StartGameInstance()
        1. UEngine::Browse()
          1. UEngine::LoadMap()

UEngine::LoadMap()에서

  1. 기존 맵이 있다면 기존 게임 오브젝트 제거 (단, SeamlessTravel에선 특정 액터의 Outer만 변경 가능)
  2. 맵 패키지(.umap) 로드.
    1. UWorld 클린업
    2. 패키지에 포함된 액터 로드. AActor::PostLoad까지 진행.
    3. 패키지에 포함된 액터들의 컴포넌트 로드. UActorComponent::PostLoad까지 진행.
    4. 물리, 네비게이션, AI, 오디오 등 여러 시스템 초기화.
    5. 액터 초기화(변수 및 Activate()), 액터 컴포넌트 초기화(UActorComponent::InitializeComponent())
  3. UWorld::SpawnPlayActor()
    1. (클라이언트 접속의 경우) AGameModeBase::PreLogin()
      • 접속 수락/거절
    2. AGameModeBase::Login()
      • 플레이어 컨트롤러 생성
        • APlayerController::PostInitializeComponents에서 APlayerState 생성
    3. AGameModeBase::PostLogin()
      • 폰 생성 및 빙의
        • APlayerController::OnPossess
        • APawn::Possessed
  4. UWorld::BeginPlay()
    1. AGameModeBase::StartPlay()
      1. AGameStateBase::HandleBeginPlay()
        1. GetWorldSettings()->NotifyBeginPlay()
          • 액터 반복자 순회하며 DispatchBeginPlay() 호출
          • AActor::BeginPlay()
          • UActorComponent::BeginPlay()

런타임 액터 스폰

  1. 메모리 할당, CreateDefaultSubobject로 CDO로부터 인스턴스 생성.
  2. PostIniProperties()에서 기본값으로 덮어씌우기.
  3. 컴포넌트를 월드에 등록
    • 렌더링, 물리연산은 이를 위한 컴포넌트에 의해 이루어짐.
    • PreInitializeComponents
    • PostInitializeComponents
  4. 블루프린트의 Construction Script, C++의 OnConstruction 호출
    • 액터 초기화 후, BeginPlay 전에 호출됨.
  5. BeginPlay()
    • UActorComponent::BeginPlay()
      (모두 한 틱에서 일어남)

클라이언트 접속 순서

  1. 클라이언트 -> 서버로 핸드셰이크 요청
  2. 서버의 NetDriver가 감지, 저수준(Socket) 연결
  3. AGameMode::PreLogin()에서 요청 수락 여부 결정 -> 접속 거부시 그대로 끝.
  4. AGameMode::Login()에서 해당 클라를 담당할 플레이어 컨트롤러 생성
  5. AGameMode::PostLogin()에서 플레이어 컨트롤러가 빙의될 폰 생성.
  6. 서버 -> 클라로 전달된 액터 Proxy들을 클라에서 스폰하여 구성.
Note

클라이언트가 서버가 있는 맵을 로드하는(맵으로 이동하는) 시점은 PreLogin에 성공한 뒤, Login 전으로 추정되나, 확실하지 않다. 보완 바람.

서버에서 생성된 액터가 클라이언트로 리플리케이트되는 순서

  1. 서버에서 생성(즉시)
  2. 넷 드라이버 업데이트 (NetDriver::TickDispatch), 리플리케이트 해야 되는 새 액터 발견
  3. 전파 시점 결정
    • NetUpdateFrequency 주기가 도래했는가?
    • Relevancy 조건을 충족하는가?
      • bAlwaysRelevant: 다른 조건에 상관없이 무조건 연관되어 전파되는가?
      • NetCullDistanceSquared: 클라이언트 카메라와 액터 사이의 거리가 충분히 가까운가?
      • Ownership Check, bHidden 등
  4. 전파하기로 결정됨.
  5. Actor Channel로 직렬화된 액터의 데이터(클래스, Transform등 생성에 필요한 것만) 전송.
  6. 클라이언트 수신, 역직렬화, 패킷 분석, 유효하고 새로운 액터 ID인지 체크.
  7. 클라이언트 측에서 SpawnActor 호출(프록시 액터 생성) -> 같은 틱에서 BeginPlay()까지 호출

SpawnActor로 전파되는 건 어디까지나 '이 Transform으로 이 액터를 너도 생성하라'는 정보 뿐이다. 서버에서 SpawnActor를 호출한 후 같은 틱에서 액터의 프로퍼티를 바꿨다고 그것까지 전달되지는 않는다.
물론, 생성한 액터에 있는 리플리케이트 되는 프로퍼티를 같은 틱에서 바꿨다면 그건 클라에서 SpawnActor가 호출된 뒤 다시 넷 패킷을 통해 그 액터의 프로퍼티 값을 바꾸라는 명령이 전달된다.
