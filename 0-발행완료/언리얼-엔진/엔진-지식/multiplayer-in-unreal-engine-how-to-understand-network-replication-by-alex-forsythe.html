Multiplayer in Unreal Engine How to Understand Network Replication by Alex Forsythe

서론

언리얼 엔진의 네트워크 기능은 매우 강력하다. 개발자는 클라이언트 연결, 소켓 열고 닫기, 패킷을 보내기, 직렬화, 인코딩, 바이트 정렬, 타임스탬핑, 패킷 재정렬, 라우팅, 손실 패킷에 대한 재전송(Retransmission) 등을 신경쓸 필요가 없다.

Net Mode

UWorld의 열거형 프로퍼티로, 현재 네트워크 연결 방식을 나타낸다.
스탠드얼론, 데디케이티드 서버, 리슨 서버, 클라이언트의 네 가지 종류이다.

(1) 플레이어로서 참여할 수 있는가
(2) 권한(Authority)을 가져 데이터를 변경할 권리를 가지는가
(3) 서버로서 클라이언트의 접속을 가능케 하는가
이 세 가지 조건을 기준으로 분류하면 다음과 같다.

이러한 넷 모드는 게임 인스턴스가 어떻게 시작되었느냐에 따라 다르게 설정된다.
From int main to BeginPlay 문서에서 게임 인스턴스가 만들어지고, 최초의 UWorld를 Browse하는 과정에서 URL을 통해 로드/접속한다고 설명했다. 이 때,

  • URL이 로컬 레벨 에셋의 경로 -> 넷 모드는 스탠드얼론이 되고, 혼자서만 플레이할 수 있다.
  • URL이 로컬 레벨 에셋의 경로 + ?Listen -> 넷 모드는 리슨 서버가 되고, 클라이언트가 접속할 수 있게 된다.
  • URL이 호스트의 IP 주소 -> 넷 모드는 클라이언트가 되고, 호스트에 접속을 요청하고, 승인받으면 호스트가 있는 레벨을 로드한다.

데디케이티드 서버는 플레이어로서 참여하지는 않고 로직만을 처리하는 서버용 인스턴스를 만드는 특수한 경우로, ULocalPlayer나 뷰포트 클라이언트같은 게 필요하지 않다. 이는 다른 특수한 방법을 통해서 실행할 수 있다.

언리얼 리플리케이트 시스템

서버와 서버에 접속한 클라이언트는 각각의 게임 인스턴스를 가지고, 각각 월드를 생성한다. 이후 언리얼의 리플리케이트 시스템에 의해 서버의 환경이 클라이언트로 전파되는 과정을 거친다.
이 때 사용되는 네트워크 클래스들이 UNetDriver, UNetConnection, UChannel이다.

먼저 서버는 자신의 넷 드라이버를 생성하고 클라이언트의 요청에 대한 수신 대기 상태가 된다. 클라이언트는 자신의 넷 드라이버를 생성하고 서버에 연결을 요청하고, 이것이 수락되면 서버와 자신 간의 연결을 관리하는 UNetConnection 객체를 생성하게 된다.
클라이언트는 서버랑만 연결되므로 하나의 넷 커넥션을 가지며, 서버는 여러 클라이언트 각각에 대해 넷 커넥션 객체를 가지고 있게 된다.

UNetConnection에는 어떤 데이터를 주고 받을 것이냐는 목적에 따라 다양한 UChannel들이 존재한다. 컨트롤러의 조작에 대한 데이터를 주고받는 UControlChannel, 보이스 채팅을 위한 UVoiceChannel, 리플리케이트되는 액터들을 위한 UActorChannel들이 대표적이다.

이 구조를 시각적으로 나타내면 다음과 같다.

액터의 리플리케이트 과정

서버가 bReplicates 플래그가 켜진 액터를 생성하면, 여러 조건(특히, Relevancy가 있는지)을 따져 부합하는 경우 해당 액터를 리플리케이트하기 위한 액터 채널을 각각의 넷 커넥션에 생성하게 된다.
액터의 생성 정보(어떤 클래스를 어떤 Transform으로 생성하는가?)가 해당 채널을 통해 전달되어 클라이언트에서도 액터를 생성하게 되며, 이 때 클라이언트에서 생성된 권한 없는 액터를 Proxy라고 부른다.

Proxy의 생명주기는 서버의 권한 있는 액터의 생명주기를 따르므로, 서버에서 액터가 제거되면 이 역시 즉시 액터 채널을 통해 전달되어 클라이언트에서도 제거된다.

프로퍼티 리플리케이트, RPC

액터 채널이 열려있는 액터에 대해 두 가지 방식으로 소통이 가능하다. 첫 번째는 UPROPERTY 매크로와 라이프타임이 지정된 replicapable 프로퍼티의 데이터를 서버 -> 클라로 보내는 것이고, 두 번째는 함수를 '호출하라'는 요구를 주고받는 것(RPC, Remote Procedure Call)이다.

프로퍼티 리플리케이트와 Server RPC, NetMulticast RPC는 서버 -> 클라로만 보낼 수 있다. Client RPC는 클라이언트가 서버로 함수 호출을 요구하는 유일한 방법인데, 이를 트리거하는 것은 클라이언트의 플레이어 컨트롤러가 소유하고 있는(PlayerController, PlayerState, 빙의된 폰이 대표적) 액터만 가능하다.

  • Server RPC는 클라이언트에 의해 트리거되며, 해당 함수를 서버의 권한있는 액터가 호출하도록 한다.
    • WithValidation 옵션을 지정하면 _Validate
  • Client RPC는 서버에 의해 트리거되며, 해당 함수를 클라이언트의 프록시가 호출하도록 한다.
  • NetMulticast RPC는 서버에 의해 트리거되며, 해당 함수를 (리슨 서버를 포함한) 모든 인스턴스의 권한 있는 액터와 프록시들이 호출하도록 한다. 이 때 NetCulling등에 걸려 연관성이 없는 클라이언트가 있다면 제외된다.

Ownership

언리얼 네트워크에서 가장 중요한 개념은 소유권(Ownership)이다.
모든 언리얼 오브젝트는 다른 언리얼 오브젝트를 소유(Own)할 수 있다. 서버 입장에서, 클라이언트가 접속해 UNetConnection을 통한 연결이 구성되면, UNetConnection은 그 자체로 클라이언트를 나타내고, 클라이언트의 조작과 제어를 담당하는 APlayerController를 UNetConnection이 소유한다고 볼 수 있다.
또 APlayerController는 APlayerState와 자신이 빙의하는 APawn 등을 소유하게 된다.

서버와 클라간의 소통 창구는 UNetConnection에 있는 네트워크 채널들이 유일한 상황에서, 이런 관계를 가지게 되기 때문에, 권한이 있는 서버에서 클라로 전파하는 건 새 채널을 열어서 수행할 수 있지만 클라에서 역으로 서버로 요구를 보내는 건 (새 채널을 열 권한이 없기 때문에) 이미 존재하는 UNetConnection을 통해서, 즉 APlayerController와 컨트롤러가 소유한 액터들을 통할 수밖에 없다.

Note

실제로 액터의 GetNetOwningPlayer()를 호출하면 Owner를 타고타고 올라가며 연결된 커넥션을 찾아가게 된다.

Relevancy

액터에 bReplicates 플래그가 켜져있다고 반드시 전파되는 것은 아니다. 가장 중요한 조건은 Relevancy를 가지는 것이다.
서버의 넷 드라이버는 리플리케이트 되는 액터나 프로퍼티의 변화를 감지하고, 갱신 주기가 왔다면 각각의 커넥션을 통해 Relevancy가 있는지 체크하게 된다. 보통은 클라이언트의 카메라가 멀리 있고, 그 클라이언트에 영향도 주지 못 한다면 NetCulling으로 일정 거리 이상일 때 전파하지 않는 기능으로 사용한다.
AGameStateBase, APlayerState와 같은 중요한 Info 게임 오브젝트들은 당연히 bAlwaysRelevant 플래그가 켜져있어 반드시 연관성이 있는 것으로 처리된다.
bNetUseOwnerRelevancy 플래그로 자신이 아닌 자신의 오너를 기준으로 연관성을 검사하는 등 다양한 옵션을 제공한다.

일반적인 상황(소유하거나 Instigating한 액터가 아니고, 다른 특별한 플래그가 세워지지 않은 경우)에선 다음 원칙에 따라 전파가 결정된다.

  1. Hidden 플래그가 켜져있고 루트 컴포넌트의 콜리전이 꺼져있음 -> False
  2. NetCullDistance 밖에 있음 -> False

기본 로직은 하드코딩되어 있으며, AActor::IsNetRelevantFor() 함수를 오버라이딩하여 커스텀 규칙을 설정하는 것도 가능하다.

Update Frequency, Priority

연관성이 있어 전파가 결정된 경우, 마지막으로 넷 업데이트 주기와 우선순위에 따라 언제, 어떤 순서로 전파가 이루어질 지 결정하게 된다.
Update Frequency는 말 그대로 액터의 변화를 초당 몇 번 전파할 것이냐이며, Priority는 넷 대역폭이 한 번에 모든데이터를 전달할 수 없을 때 무엇을 우선할 것이냐를 결정한다. (물론, 하나의 채널 안에서는 순서대로 전파된다) 우선순위는 얼마나 오랫동안 전파되지 않았는지, 거리가 가까운지 등 여러 요소가 복합적으로 고려되어 계산된다.

주의할 점은 실제 네트워크 환경은 매우 다양한 변수, 지연, 대역폭을 가지므로 단순히 주기를 높힌다고 부드럽게 전파된다는 보장은 없다는 것. 예컨대 연속적인 변화를 전파해야 한다면 가장 좋은 방법은 일정 간격의 포인트에 대해서만 전파하고 그 사이는 클라이언트가 직접 보간하게 하는 것이다.

Role

  • ROLE_None : 전파되지 않는 레벨 액터.
  • ROLE_Authority : 서버의 권한 있는 액터. HasAuthority()가 true를 반환.
  • ROLE_AutonomousProxy : 클라의 제어 권한이 있는 프록시. 컨트롤러와 컨트롤러가 빙의한 폰이 여기에 해당. IsLocallyControlled()가 true를 반환.
  • ROLE_SimulatedProxy : 클라의 제어 권한이 없는 프록시. 다른 클라이언트의 폰이나 전파되는 모든 액터가 여기에 해당.

네트워크를 다루는 팁

방어 코드

어떤 함수가 서버에서만, 혹은 로컬 플레이어일 때만 호출해야 한다면 다음과 같이 방어 코드를 두는 것이 좋다.

checkf(HasAuthority(), TEXT("This code should only run on the server!"));

혹은 함수의 이름을 명확하게 짓는 것도 도움이 된다.

void ServerRPC_DoSomething();
void AuthDoSomething();

전파 시점과 주체 고려하기

어떤 로직이냐에 따라 저마다 적합한 전파 시점과 주체를 가진다.
예컨대 클라이언트가 총을 쏜다면, 총을 쏘는 키를 누르는 건 클라이언트가, 총알을 스폰하고 라인트레이스를 후 데미지를 입히는 로직은 서버가, 피가 터지는 이펙트는 모든 클라이언트가 보아야 한다.
