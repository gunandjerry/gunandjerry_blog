언리얼 비헤이비어 트리

언리얼 BT 기초

언리얼 BT는 Blackboard, Behavior Tree 두 개의 에셋으로 이루어진다.
Blackboard는 BT에서 사용하기 위한 변수 컨테이너같은 역할을 한다. BT가 동작할 때 결정을 내리기 위한 여러 플래그나 정보, 대상을 설정하기 위한 포인터 등이 필요한데, 이것들을 저장하는 역할이다.

기본적으로 표준적인 BT 설계 방식과 유사하나, 언리얼만의 차별점이 있다.

표준 BT와의 공통점

  • 위에서 아래, 왼쪽에서 오른쪽으로 진행한다. 노드 우상단의 숫자를 통해 동작 순서를 알 수 있다.
  • BT의 기본 동작을 구현하는 Compsite 노드인 Selector와 Sequence가 제공된다.

표준 BT와의 차이점

  • 이벤트 주도형으로, 외부에서 이벤트를 통해 BT가 요구하는 정보를 갱신시키므로, 매 프레임 BT를 통해 불필요한 작업을 할 필요가 없다.
  • 조건 분기 노드는 없다. 대신 Composite 노드에 Decorator를 ‘부착’하는 방식으로 조건 검사를 진행한다.
  • 구체적인 로직이 필요한 동작은 Task 노드 하나로 묶이며, 블루프린트 또는 UBTTaskNode를 상속한 노드 클래스로 정의할 수 있다.
  • Composite 노드군에 Simple Parallel 노드라는 병렬 동작을 위한 노드가 추가되었다.
  • Composite 노드에 부착할 수 있는 Service 노드라는 특수한 노드가 추가되었다.
  • 병렬 프로세싱 로직이 완전히 다르다. 아래에서 자세히 설명한다.

동시 발생 비헤이비어

표준적인 BT에서는 병렬 Composite 노드를 사용해 여러 비헤이비어를 동시에 작동시킨다. 이런 유형의 노드는 모든 자손을 병렬적으로 동시에 동작시키며, 이 중 하나가 종료될 경우의 처리는 특수 규칙으로 정의한다.

Note

여기서 설명하는 ‘동시 진행’이 반드시 멀티 스레딩일 필요는 없다. 어디까지나 개념적인 방식으로, 시분할일 수도 있다.

반면 언리얼 엔진은 Simple Parallel 노드와 Service라는 특수 노드, 그리고 Decorator의 관찰자 중단(Observer Aborts)를 통해 이를 구현하고 있다.

Simple Parallel 노드

Simple Parallel 노드는 하나의 Task 노드를 포함하여 두 개의 자식을 가지며, ‘Task를 수행하는 동안 나머지 자식 노드도 수행한다’로 이해하면 된다. 일반적으로 Task 노드 자식은 주요 행동이고, 나머지 자식은 A가 완료되기까지 기다리는 도중의 부가적인 행동으로 정의된다.

Service 노드

부착된 Composite 노드가 동작하는 동안(Running 상태) 지정된 시간마다 일정한 로직을 콜백으로 호출하는 노드. 예컨대 AI 폰이 적을 쫓는 BT가 동작하는 동안 실시간으로 어느 적이 최적의 추격 대상인지 갱신하기 위해 Service 노드를 사용할 수 있다.

Decorator의 Observer Aborts

Decorator 노드는 설정된 조건이 false를 반환할 경우 즉시 false를 반환하고 빠져나오게 되어있다. 일반적으로 표준 병렬 노드는 지속적으로 조건을 검사하며 조건에 부합할 때만 동작하는 시퀀스를 실행시키기 위해 사용되기도 하는데, 언리얼 BT는 이벤트 주도 방식인 만큼 그렇게 쓰지 말고 그냥 외부에서 이벤트로 Blackboard의 값을 갱신시키고 Decorator에 의해 태스크를 중단하도록 하면 된다.

이벤트 주도 BT

이벤트 주도형 BT라는 것은 BT의 동작에 필요한 정보들이 이벤트를 통해 갱신되고, BT는 그 갱신된 정보에 따라 필수적인 분기만을 거쳐 동작하여 매 프레임 불필요한 조건 검사를 할 필요가 없음을 의미한다.

표준 BT 모델은 기본적으로 경로에 있는 모든 노드를 순차적으로 거치기 때문에 비용이 크다는 문제가 있었다. 하지만 이벤트 주도 방식을 사용하면 가장 비용이 큰 조건 검사를 모두 외부에 맡기고 핵심적인 분기만을 거쳐 바로바로 필요한 동작을 실행하도록 설계하는 게 가능하다.

예컨대 위와 같은 언리얼 문서 샘플 BT를 보면, Chase Player 시퀀스에 들어가기 전에 Has Line of Sight 플래그(Blackboard에 있는 변수)가 켜져있는지 Decorator를 통해 체크하고, 그렇지 않다면 아예 하위 트리로 들어가지 않는다.

C++로 Has Line of Sight 플래그 켜기

그렇다면 저런 모양의 BT가 있을 때, 우린 외부에서 시야가 있는지 없는지를 대신 판별해서 값을 갱신시켜줄 것이다.
해당 BT를 사용하는 AIController의 상속 클래스를 다음과 같이 작성했다.

언리얼이 BT와 Blackboard 에셋을 구분한 것은 아래와 같은 설계를 위해서다.

  1. AIController 및 다른 클래스에서 특정한 이벤트가 발생했을 때(예컨대, AIController가 특정한 액터를 ‘볼 수 있을 때’ 호출되는 AIPerception 컴포넌트의 이벤트) Blackboard의 특정 키의 값을 바꾼다.
  2. BT는 AI의 동작만을 정의하며, Blackboard에 기록된 값들에 따라 분기한다.

다음은 등록된 Pawn에 빙의함과 동시에 등록된 BT를 동작시키고, AIPerception에 의해 Player 태그를 가진 액터를 볼 수 있을 때 등록된 Blackboard의 LineOfSight와 EnemyActor의 값을 변경시키는 AIController의 예시이다.

SampleBTAIController.h

#pragma once

#include "CoreMinimal.h"
#include "AIController.h"

#include "Perception/AIPerceptionTypes.h"

#include "SampleBTAIController.generated.h"

UCLASS()
class BTSAMPLE_API ASampleBTAIController : public AAIController
{
	GENERATED_BODY()
	
public:
	// 등록된 BT를 OnPossess 시점에 동작
	UPROPERTY(EditDefaultsOnly, Category = "AI")
	class UBehaviorTree* BehaviorTree;
private:
	UPROPERTY(VisibleAnywhere, Category = "AI")
	class UAIPerceptionComponent* AIPerceptionComponent;
	UPROPERTY(VisibleAnywhere, Category = "AI")
	class UAISenseConfig_Sight* SightConfig;

public:
	ASampleBTAIController();

	UFUNCTION()
	void OnTargetPerceptionUpdated(AActor* Actor, FAIStimulus Stimulus);
protected:
	virtual void BeginPlay() override;
	virtual void OnPossess(APawn* InPawn) override;
};

SampleBTAIController.cpp

#include "SampleBTAIController.h"

#include "BehaviorTree/BehaviorTree.h"
#include "BehaviorTree/BlackboardComponent.h"

#include "Perception/AIPerceptionComponent.h"
#include "Perception/AISenseConfig_Sight.h"

#include "Engine/Engine.h"

// 생성자에선 AIPerception 컴포넌트를 등록하고 시각 감각 정보를 사용하도록 지정
ASampleBTAIController::ASampleBTAIController()
{
	AIPerceptionComponent = CreateDefaultSubobject<UAIPerceptionComponent>(TEXT("AIPerceptionComponent"));
	SetPerceptionComponent(*AIPerceptionComponent); // 폰의 AIPerception(감각 컴포넌트)로 등록

	SightConfig = CreateDefaultSubobject<UAISenseConfig_Sight>(TEXT("SightConfig"));
	SightConfig->SightRadius = 1000.0f;
	SightConfig->LoseSightRadius = 1200.0f;
	SightConfig->PeripheralVisionAngleDegrees = 90.0f;
	SightConfig->DetectionByAffiliation.bDetectEnemies = true;
	SightConfig->DetectionByAffiliation.bDetectFriendlies = true;
	SightConfig->DetectionByAffiliation.bDetectNeutrals = true;

	AIPerceptionComponent->ConfigureSense(*SightConfig);
	// AIPerception은 여러 감각을 동시에 감지할 수 있는데, 이 중 가장 주된 감각이 되는 게 Dominant Sense
	AIPerceptionComponent->SetDominantSense(*SightConfig->GetSenseImplementation());
}

// BeginPlay에선 AIPerception이 감각 감지에 성공했을 때 호출될 대리자를 등록
void ASampleBTAIController::BeginPlay()
{
	Super::BeginPlay();

	if (AIPerceptionComponent)
	{
		AIPerceptionComponent->OnTargetPerceptionUpdated.AddDynamic(this, &ASampleBTAIController::OnTargetPerceptionUpdated);
	}
}

// Actor가 Player 태그를 가지고 있으면, Blackboard를 가져와 EnemyActor, HasLineOfSight 키의 값을 바꿈
void ASampleBTAIController::OnTargetPerceptionUpdated(AActor* Actor, FAIStimulus Stimulus)
{
	if (Actor && Actor->ActorHasTag("Player"))
	{
		UBlackboardComponent* BlackboardComp = GetBlackboardComponent();
		if (BlackboardComp)
		{
			if (Stimulus.WasSuccessfullySensed()) // 감지에 성공했다면
			{
				BlackboardComp->SetValueAsObject("EnemyActor", Actor);
				BlackboardComp->SetValueAsBool("HasLineOfSight", true);
			}
			else
			{
				BlackboardComp->ClearValue("EnemyActor");
				BlackboardComp->ClearValue("HasLineOfSight");
			}
		}
	}
}

// 빙의와 동시에 BT 동작
void ASampleBTAIController::OnPossess(APawn* InPawn)
{
	Super::OnPossess(InPawn);

	RunBehaviorTree(BehaviorTree);
}

※ 위 코드는 시각적으로 감지되지 않는 순간 곧바로 EnemyActor가 풀려버린다. 따라서 일반적으로 타이머를 넣어 n초 동안은 EnemyActor가 담긴 상태가유지되도록 처리한다.

이제 위 AIController를 열어 BT를 등록하고, Pawn이 해당 AIController를 사용하도록 설정하면 된다.

언리얼 BT의 노드들

Root Node

BT가 시작되는 지점.

Composite Node

분기가 실행되는 기본 규칙을 정의하는 노드들.

Selector Node

여러 분기 중 하나를 선택하는 노드. 자식 노드를 순회하여 최초로 true를 반환(즉, 최초로 성공한 자식 노드를 발견)받을 때 true를 반환하고, 모든 자식 노드를 순회해도 true를 반환받지 못 하면 false를 반환한다.

Sequence Node

순차적으로 일련의 프로세스를 동작시키는 노드. 자식 노드를 순회하며 true를 반환받으면 다음 자식으로 진행하고, false를 반환받으면 순차 진행을 멈추고 false를 반환한다. 모든 자식을 성공적으로 순회하면 true를 반환한다.

Simple Parallel

하나의 Task 노드를 포함해 두 개의 자식 노드를 가지는 병렬 동작 노드. Task 노드가 동작하는 동안 나머지 노드도 동시에 동작한다.

Task Node

일련의 로직을 실행하는 노드. 블루프린트 또는 Cpp로 상속(BTTaskNode(UBTTaskNode))하여 제작한다.

Decorator Node

표준 BT의 조건 분기 노드에 해당한다. Composite 노드에 부착하여 조건 검사를 진행해 통과할 때만 Composite 노드가 동작(즉, 서브 트리를 순회)하고, 통과하지 못 하면 그 즉시 바로 실패를 반환하고 빠져나온다. 이미 서브 트리를 순회하는 상태라도 조건이 false가 되는 순간 빠져나온다는 점에 주의하자.
Composite 노드를 우클릭하고 Add Decorator → Blueprint를 선택하고 Blackboard의 키에 대해 쿼리하는 게 가장 보편적인 방식이고, 블루프린트 또는 Cpp로 상속하여 별도의 조건 로직을 만들 수도 있다.

Service Node

부착된 Composite 노드가 동작하는 동안(Running 상태) 지정된 시간마다 일정한 로직을 콜백으로 호출하는 노드. 예컨대 AI 폰이 적을 쫓는 BT가 동작하는 동안 실시간으로 어느 적이 최적의 추격 대상인지 갱신하기 위해 Service 노드를 사용할 수 있다.
블루프린트 또는 Cpp로 상속하여 별도의 콜백 로직을 만들 수 있다. Service가 주기적으로 콜백을 호출하는 건 어디까지나 자신이 부착된 Composite 노드가 동작하는 중(Running 상태)일 때만임을 명심하자.

Note

Task는 Cpp로 작성하는데 큰 불편함이 없는데, Decorator와 Service는 Cpp로 작성하기 굉장히 불편하다.

Blackboard

BT에서 사용할 정보들을 키-값 쌍 형태로 저장하는 컨테이너.
Blackboard의 키의 Details에서는 다음과 같은 설정을 할 수 있다.

  • Entry Name : 키의 이름. Blackboard Key Selector를 통해 FString으로 찾을 수 있으므로 매우 중요하다.
  • Instance Synced : 체크하면 블랙보드의 모든 인스턴스(해당 블랙보드를 사용하는 BT마다 생성된다)를 전역적으로 동기화한다.
  • Key Type - Base Class : 해당 클래스만 담을 수 있게 제한할 수 있다.

열거형 Key의 경우 Key Type 세부설정을 열어서 다음 설정을 추가로 할 수 있다.

  • Enum Type : Enumeration 에셋을 지정하여 Enum 타입을 특정한다.
  • Enum Name : C++ 코드로 정의한 Enum 타입을 이름으로 찾아서 특정한다. 이름으로 찾는데 성공하면 Enum Type에 지정한 에셋보다 우선된다.
  • Is Enum Name Valid : 열거형 값을 DisplayName으로 오버라이드 할 수 있는지 없는지 여부.
