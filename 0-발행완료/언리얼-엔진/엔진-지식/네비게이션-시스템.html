네비게이션 시스템

길찾기 AI의 분류

많이 사용하는 길찾기 AI는 크게 두 가지 방식으로 나뉜다.

  1. Waypoints : 길 곳곳에 웨이포인트를 찍어놓고 이들을 노드로 삼아 길찾기 알고리즘을 동작시키는 방식.
  2. Navigation Mesh : 볼륨 내의 모든 콜리전 지오메트리를 체크하여 이동 가능한 영역을 담는 메쉬를 생성하고, 그 메쉬를 이루는 각각의 폴리곤 또는 폴리곤의 꼭지점을 노드로 삼아 길찾기 알고리즘을 동작시키는 방식.

Waypoints 방식은 구현이 간단하지만 길에 웨이포인트를 찍는 전처리 단계가 필요하고, 웨이포인트들 간의 직선 이동만 가능하므로 자연스러운 경로를 생성하기 어렵다는 단점이 있었다.
또 찍어둔 웨이포인트의 개수만큼 노드가 되므로 큰 맵에서의 부하도 무시할 수 없다.

반면 NavMesh 방식은 다각형(주로 삼각형 폴리곤)을 기반으로 경로를 생성하므로 훨씬 자연스럽고 부드러운 경로를 생성할 수 있다.
무엇보다 웨이포인트를 찍는 전처리 단계가 필요하지 않으므로 다양한 환경에 자동으로 대응할 수 있어 편리하다.

일반적으로 웨이포인트 방식을 사용하려면 자연스러운 경로를 구현하기 위해 많은 수의 웨이포인트들이 생성되어야 하기 때문에, 경우에 따라선 NavMesh 방식의 가성비가 더 좋을 수 있다.

구체적으론 Simple Stupid Funnel Algorithm등 다양한 방법을 사용한다(고 한다).

네비게이션 시스템 적용하기

Navigation Mesh Bounds Volume

기본적으로 이 바운딩 볼륨의 안에서 NavMesh가 생성된다. 그 시점은 이 액터가 레벨에 추가되거나 크기가 조정되는 즉시이며, 기본 프로젝트 설정으로는 여기서 만들어진 메시를 그대로 가져다 사용하게 된다.

NavMeshBoundsVolume 액터를 추가하면 자동으로 NavMesh 설정을 위한 RecastNavMesh-Default 액터도 추가되며, 이 액터의 Details에서 다음과 같은 설정을 할 수 있다.

  1. Display 섹션에서 다양한 기즈모를 켜고 끌 수 있다. 예컨대, Draw Triangle Edges는 NavMesh의 폴리곤이 어떻게 분할되었는지 보여준다.
  2. Display 섹션의 Draw Offset은 생성된 NavMesh의 ‘이동 가능한 영역’을 나타내는 칠의 Z축 Offset인데, NavMesh가 콜라이더를 간소화하여 생성되다보니 다른 메쉬와 겹쳐 알아보기 힘든 경우엔 올리도록 하자.

Pawn이 NavMesh를 통해 움직이게 하기

일단 NavMesh가 생성되면 사용법은 매우 간단하다.

// 본인이 속한 NavMesh 내에서 이동 가능한 위치를 랜덤하게 찾는다.
// 본인의 위치 기준으로 555cm 반경 내
FVector randomPos;
GetRandomPositionInNavMesh(this->GetActorLocation(), 555, randomPos);

// AI 컨트롤러의 MoveToLocation을 사용해 해당 좌표로 이동시킨다.
AAIController* controller = Cast<AAIController>(this->GetController());
controller->MoveToLocation(randomPos);

이번엔 조금 활용해서 특정 위치(이 경우엔 타겟의 위치)로 이동이 가능한 경우엔 그리로 가고, 아니면 랜덤 위치로 패트롤하도록 해보자.

#include "NavigationSystem.h"
#include "Navigation/PathFollowingComponent.h"

...

FVector targetPos = target->GetActorLocation();

// 현재 월드의 네비게이션 시스템 가져오기
auto NavSys = UNavigationSystemV1::GetNavigationSystem(GetWorld());

// 이동 가능 여부를 쿼리하기 위한 쿼리 객체 생성
FPathFindingQuery query;
FAIMoveRequest moveReq; // 이동 시도 쿼리
moveReq.SetAcceptanceRadius(3); // 3cm 이내에 도달이 가능하면 true
moveReq.SetGoalLocation(targetPos);
// 컨트롤러에게 쿼리 시뮬레이션 요청
controller->BuildPathfindingQuery(moveReq, query);

// 쿼리 결과 얻어오기
FPathFindingResult result = NavSys->FindPathSync(query);
if (result.Result == ENavigationQueryResult::Success)
{
	// 쿼리에 성공 = targetPos로 이동 가능
	controller->MoveToLocation(targetPos);
}
else
{
	// 쿼리 실패 -> 랜덤 위치로 패트롤
	auto moveResult = controller->MoveToLocation(randomPos);
	// 랜덤 위치에 도달했으면 랜덤 위치를 변경
	if (moveResult == EPathFollowingRequestResult::AlreadyAtGoal)
	{
		GetRandomPositionInNavMesh(this->GetActorLocation(), 555, randomPos);
	}
}

네비게이션 링크 프록시(Navigation Link Proxy)

이어지지 않은 NavMesh 영역을 이어주는 볼륨. NavMesh 사이의 공간을 연결하는 다리를 만들거나, 목적지로 가는 이어진 경로가 없을 때 목적지를 향해 플랫폼에서 뛰어내리거나 점프하는 등의 동작을 하도록 만든다.

두 영역을 연결하는 다리 만들기

Nav Link Proxy 액터를 레벨에 배치하고, PointLinks[0]의 Left, Right 핸들을 각각 이을 영역의 양단에 배치한다.

Note

Nav Link Proxy로 연결이 가능하면(두 핸들이 NavMesh 표면과 성공적으로 연결된 경우) 짙은 초록색의 각진 화살표가 생성되므로 참고하도록 하자.

Nav Link Proxy에 의해 연결된 경로는 Nav Mesh에 의한 내비게이션 데이터가 없어도 이동할 수 있다.

프록시 액터의 Details 창에서 다음 설정을 할 수 있다.

  • Point Links - 인덱스 - Direction 드롭다운 항목에서 양방향 또는 한 방향으로 이동 가능하게 설정할 수 있다. 예컨대 Right to Left로 설정한 경우, AIController는 반드시 Right 핸들에서 Left 핸들로만 이동한다.

액터가 플랫폼에서 떨어지게 하기

핸들을 높은 플랫폼 위에 하나, 그 밑의 바닥에 하나 배치하면 뛰어 내리는 선택을 할 수 있다.

액터가 플랫폼으로 뛰어오르게 하기

이번엔 Nav Link Proxy의 Smart Link를 사용해 한 플랫폼에서 다른 플랫폼으로 점프하게 해보자. 이를 위해선 Nav Link Proxy 클래스를 상속해서 특정 이벤트가 발생했을 때 특정 액션을 호출하는 블루프린트를 만들어야 한다. (안타깝게도 이벤트 함수가 가상함수가 아니라 블루프린트로만 구현할 수 있게 되어있다)

Smart Link란?

일반 Nav Link보다 지능적으로 복잡한 로직을 처리해주는 링크. 특정 조건에 따라 연결을 활성화하거나 비활성화할 수 있다.

예컨대 문이 열려있을 때만 링크를 활성화하여 지나갈 수 있다고 판단하게 하던가, 플랫폼의 움직임에 따라 동적으로 링크된 경로를 업데이트하는 등.

위 블루프린트는 단순히 아치 형태로 속도를 넣어 마치 뛰는 것처럼 만드는 것이다.

필요하다면 점프 함수를 직접 구현하고 이를 호출하는 방식을 쓰자.

Note

NavMeshBoundsVolume 액터와 함께 생성된 RecastNavMesh-Default의 Generation 섹션의 Generate Nav Links를 체크하면 자동으로 적절한 위치에 Nav Link Proxy들을 생성해준다.

바로 아래에 있는 Nav Link Jump Down Config에서 자동으로 생성되는 Link Proxy의 생성 조건 등을 설정할 수 있다.

런타임에 내비게이션 메시 생성하기

Project Settings - Navigation Mesh - Runtime 섹션의 Runtime Generation 드롭다웃 항목은 각각 다음과 같다.

Static Nav Mesh가 미리 생성되어 레벨과 함께 저장됨. 런타임에 변경 불가능.
Dynamic Nav Mesh가 미리 생성되어 레벨과 함께 저장되거나, 혹은 런타임에 실시간으로 만들어질 수 있음. 런타임에 데이터가 변경되어 영향을 받은 일부 타일이 업데이트될 수 있음.
Dynamic Modifiers Only Nav Mesh가 미리 생성되어 레벨과 함께 저장되나, 런타임에 Navigation Modifier를 통해 기존 Nav Mesh를 제한적으로 수정할 수 있음. 런타임에 새로운 Nav Mesh 표면이 생성되지는 않는다.
Note

Pawn을 상속한 액터의 경우 기본적으로 동적 네브 메쉬 생성에 영향을 주지 않도록 되어있다. Pawn - Can Affect Navigation Generation을 체크하면 그 때부터 영향을 주는데, 이러면 폰이 길찾기를 할 때 자기 자신을 장애물로 취급해 탭댄스를 추기 때문에, 일반적으로 움직일 땐 설정을 끄고 멈춰있을 때만 설정을 켜는 식으로 사용한다.

Note

런타임에 실시간으로 생성되고 있는 메쉬 표면의 해당 타일은 Navigation Show(P) 상으로 빨간색으로 칠해진다.

Navigation Modifier

AIController는 항상 Nav Mesh를 통해 이동할 때 최적의 경로를 찾아 이동한다.
이 때 특정 영역에 특정한 비용 가중치를 매기고 싶다면 Navigation Modifier를 사용한다.

Navigation Modifier Volume

볼륨 내에 네비게이션 가중치를 먹이는 볼륨 액터.
영역 클래스(Area Class)를 사용해 모디파이어 볼륨 내 네비게이션의 이동 비용에 가중치를 입히거나 에이전트가 해당 영역에 들어갈 때 적용되는 초기 비용인 ‘고정된 영역 진입 비용(Fixed Area Entering Cost)’ 등을 정의할 수 있다.

Area Class

특정 네비게이션 영역에 먹일 가중치를 정의하는 클래스. Nav Modifier Volume 액터의 Area Class 프로퍼티에 넣어서 적용할 수 있다.
기본적으로 NavArea_Default, LowHeight, Null, Obstacle이 만들어져 있으며 각각의 속성은 다음과 같다.

NavArea_Default 기본 설정. 이동 가능한 구역.
NavArea_LowHeight 숙여야지만 지나갈 수 있는 낮은 높이 구역. 기본적으로 별도의 조치가 없으면 AIController가 아예 지나가지 않는다.
NavArea_Null 지나갈 수 없는 구역.
NavArea_Obstacle 장애물이 있어 지나가는 비용이 높은 구역. 기본적으로 별도의 조치가 없어도 AIController가 비용을 계산하여 이득이면 지나간다.

Area Class 직접 정의하기

이번엔 Area Class를 직접 만들어보자. NavArea 클래스를 상속한 블루프린트 또는 Cpp 클래스를 생성하여 만들 수 있는데, 기본적인 비용 설정은 이미 제공되므로 더 복잡한 구현이 필요한 게 아니라면 블루프린트로 만들자.
블루프린트의 디테일 창의 Nav Area 섹션에서 다음 설정을 할 수 있다.

  • Default Cost : 경로 이동 비용 (기본 1.0)
  • Fixed Area Entering Cost : 해당 영역에 진입할 때 드는 비용 (기본 0)
  • Draw Color : 에디터의 네비게이션 영역 기즈모(P)의 칠 색상

Navigation Query Filter

Note

Pawn 블루프린트의 Default - Filter Class에 넣으라는데 그런 항목이 없다. 혹시 Deprecated?

Nav Modifier Volume이 볼륨 내 특정 영역에서 가중치를 먹이는 거였다면, 이건 AIController나 Nav Mesh 자체에 필터를 입혀서 길찾기를 수행하도록 만드는 것이다.

사용법은 간단하다. 이 역시 가장 기본인 비용 설정은 이미 제공하므로 복잡한 구현이 필요하지 않다면 NavigationQueryFilter를 상속한 블루프린트로 생성하자.

블루프린트를 열고 Filter 섹션에서 Areas를 추가한 뒤, Area Class를 넣는다.

다른 에이전트를 회피하기

언리얼은 이동중인 다른 에이전트를 회피할 수 있는 두 가지 기능을 제공한다.

  1. Reciprocal Velocity Obstacles, RVO : 각 에이전트가 현재 속도로 등속운동한다고 가정하고 인근 에이전트를 회피하려고 시도하게 되며, 원래 방향(타겟으로 향하는 최적의 방향)과 가장 가까운 회피 가능한 방향을 고르게 된다.
  2. Detour Crowd Manager : 적응형 RVO 샘플링이란 걸 사용해서 기존 RVO보다 훨씬 고퀄리티의 회피 성능을 보여준다. 샘플링 패턴, 최대 고려 에이전트 수, 반경 등을 세세하게 설정 가능하다. DetourCrowd AI Controller 클래스에 포함되며 어떤 폰 클래스도 사용할 수 있다.

RVO 적용하기

폰의 Character Movement 컴포넌트의 Character Movement: Avoidance 섹션에서 Use RVOAvoidance만 체크하면 된다.
그 밑의 Avoidance Consideration Range는 감지 반경.

Detour Crowd Manager 사용하기

폰의 AI Controller Class 항목(기본값으로 AIController로 되어있음)을 DetourCrowdAIController로 바꾼다.
Project Settings - Crowd Manager에서 관련 설정을 할 수 있다.

Note

Use RVOAvoidance도 체크하고 AI Controller Class도 DetourCrowdAIController로 바꾸면 블루프린트 충돌 에러난다.

네비게이션 인보커 사용하기

런타임에 실시간으로 에이전트 주변에 해당 에이전트를 위한 네비 메쉬를 생성하는 컴포넌트.
주로 아래와 같은 경우에 사용한다.

  1. 맵이 너무 대규모라 NavMesh를 미리 빌드해놓는 게 오히려 실용적이지 못 한 경우
  2. AI가 보지 못 하는 영역을 포함해 항상 최적의 경로만 찾는 것이 어색한 경우

사전 설정

우선 실시간 NavMesh 생성이 가능해야 하므로

  1. Project Settings - Navigation Mesh - Runtime - Runtime Generation을 Dynamic으로 바꿔준다.
  2. Navigation System - Navigation Enforcing - Generate Navigation Only Around Navigation Invokers를 체크한다.
  3. 인보커가 Nav Mesh를 생성할 수 있는 영역을 NavMeshBoundsVolume을 레벨에 배치하여 설정한다.

인보커 사용하기

Pawn 블루프린트를 열고 Add Component - Navigation Invoker를 추가한다.
인보커의 Navigation 섹션에서 다음 설정을 할 수 있다.

  1. Tile Generation Radius : Nav Mesh가 생성되는 액터 주변 반경
  2. Tile Removal Radius : 생성되었던 Nav Mesh가 제거되기 시작하는 액터로부터의 거리 반경

네비게이션 메시 최적화하기

레벨에 배치된 모든 액터는 자체적인 충돌 지오메트리 (Collision Geometry)를 가질 수 있으며, 이들이 모여 레벨의 World Collision Geometry를 형성한다.
여기에 사용되는 기하 도형들은 기본적으로 렌더링에 사용되는 디테일한 메시보다 훨씬 단순한 형태를 사용해 성능을 최적화한다.

Nav Mesh도 이러한 World Collision Geometry로부터 생성된다. 또, 필요한 경우 런타임에 Nav Mesh의 일부를 재생성할 수 있도록 타일로 분할되어 만들어진다.
이러한 지식을 전제로, Nav Mesh의 생성 속도를 최적화할 방법을 알아보자.

복셀의 크기와 높이를 키우기

Nav Mesh는 생성될 때 복셀을 사용한다. 복셀이 작을 수록 정밀도가 높아지고 장애물과 접하는 부분의 경로 설정이 더 정확해진다.
이 복셀의 크기를 키우면 정확도를 희생하는 대신 성능을 높힐 수 있다.
Project Settings - Navigation Mesh - Generation - Cell Size와 Cell Height를 수정하면 된다.

혹은, Nav Bounds Volume 액터와 함께 추가되는 RecastNavMesh-Default 액터의 Details에서 현재 레벨의 Nav Mesh에 대해서만 복셀 크기를 지정할 수도 있다.

타일 크기 제한하기

앞서 Nav Mesh는 기본적으로 타일로 분할되어 생성된다고 했다. 하나의 타일은 여러 개의 복셀로 만들어지며, 타일이 재생성될 때 모든 복셀이 새 콜리전 정보로 재생성된다.
이는 하나의 타일이 클 수록 그 안에 들어가는 복셀이 많아지고 생성 비용이 커진다는 뜻이고, Nav Mesh를 동적으로 생성할 땐 타일이 작을 수록 필요한 영역만 재생성할 수 있으므로 효율적일 것처럼 보인다.
그러나 실제로는 하나의 타일을 생성할 때 주변에 인접한 셀들을 함께 처리하는 등 오버헤드가 발생하기 때문에, 다수의 작은 타일을 만드는 것보다 해당 영역 전체를 커버하는 하나의 큰 타일을 만드는 게 나을 수도 있다.

언리얼에선 타일의 크기는 32 ~ 128개의 복셀로 설정할 것을 권고하고 있으며 이렇게 할 때 런타임에 타일을 재생성할 때 최고의 퍼포먼스가 나온다고 한다.
예컨대 복셀의 크기를 64로 설정했다면, 타일의 크기는 2048(32 64) ~ 8192(128 64)로 설정하는 게 좋다는 뜻.

메시에 간소화된 콜리전 정보 사용하기

가장 기본적인 최적화는 충돌 다각형으로 단순한 형태를 사용하여 World Collision Geometry의 처리 비용 자체를 줄이는 것이다.

Nav Mesh 생성에 영향을 주는 오브젝트 수 줄이기

Nav Mesh가 생성될 때 고려해야 하는 오브젝트가 많을 수록 비용도 커진다. 굳이 메시를 생성할 때 고려하지 않아도 되는 오브젝트라면 Collision - Can Ever Affect Navigation 플래그를 꺼주자.

게임을 로드할 때 모든 지오메트리가 생성될 때까지 Nav Mesh가 생성되지 않도록 하기

동적으로 Nav Mesh를 생성하는 경우엔 레벨이 로드되면서 오브젝트가 하나하나 생길 때마다 Nav Mesh가 재생성될 수 있다. (아닐 수도 있다.)
따라서 로딩이 끝날 때까지는 네비게이션 시스템의 bInitialBuildingLocked 플래그를 true로 설정해 생성되지 않도록 하고, 모든 지오메트리가 로드된 뒤에 ReleaseInitialBuildingLock() 함수를 호출하여 다시 생성되도록 하자.
잠금이 해제되면 네비게이션 시스템은 새로 배치된 지오메트리에 의해 ‘dirty’ 플래그가 켜진 모든 타일을 재생성하게 된다.

Nav Mesh를 멀티 스레드로 생성하기

내비게이션 시스템에서 MaxSimultaneousTileGenerationJobsCount 프로퍼티 값을 올려 멀티 스레드로 생성하도록 하자. 이 값은 FRecastNavMeshGenerator::Init() 내의 총 작업 가능 스레드 수가 상한이다.

Static Geometry를 동적 장애물(Dynamic Obstacle)로 설정하기

액터의 Navigation - Dynamic Obstacle 플래그를 체크하면 동적 장애물이 된다.
동적 장애물은 재생성이 되어야 하는 메시 표면을 기록하여 네비게이션 시스템이 Nav Mesh를 업데이트할 때 타일 전체를 재생성하는 대신 해당 영역만 재생성하도록 최적화될 수 있다.
동적으로 움직이는 장애물에는 사용하면 안 된다.

네비게이션 메시 해상도

네비게이션 메시를 생성할 때 3가지 해상도 단계를 두고 각각 복셀의 크기를 다르게 설정할 수 있다.
Project Settings - Navigation Mesh - Nav Mesh Resolution Params에서 Low, Default, High 해상도에 따라 각각 복셀의 크기를 지정할 수 있다. (이 역시 RecastNavMesh-Default의 Details에서 현재 레벨에 대해서만 설정할 수 있다.)
Nav Modifier Volume을 배치하고 해당 볼륨 안에 포함되는 타일의 해상도를 특정한 해상도로 설정할 수 있고, 아니면 Pawn에 Nav Modifier 컴포넌트를 부착하고 Navigation - Advanced - Nav Mesh Resolution에 해상도를 지정하여 Pawn이 속한 타일의 해상도를 바꿀 수 있다.

문서에서는 성능을 최대화하기 위해 모든 해상도의 Cell Size로 타일 사이즈를 나눴을 때 나누어 떨어지는 값으로 설정할것을 권고한다.
