concept, requires

C++20에서 추가된 템플릿 메타 프로그래밍을 위한 도구.
requires 키워드로 올 수 있는 템플릿 타입의 조건을 규정하고(어떤 타입이 올 수 있는지 정의할 뿐 아니라, 매개변수에 의미론적(semantic) 제약을 가할 수도 있다), concept로 이를 캡슐화할 수 있다.

concept

캡슐화된 제약 조건 덩어리.
컴파일 타임에 결정되는 bool을 반환하거나 그러한 값으로 귀결될 수 있어야 한다.
constexpr, consteval 키워드로 bool 타입을 반환하는 컴파일 타임 함수를 만들어 사용할 수도 있고, <type_traits>, <concepts> 헤더에 정의된 제네릭 함수들을 써도 된다.

형식 제약 컨셉

템플릿 타입이 '어떤 타입'이어야 한다는 형식 제약을 규정한다.
concept 이름 = 형식 제약 꼴로 선언하며, 컴파일 타임에 bool 값을 반환하는 식이어야 한다.

#include <type_traits>

class Weapon {};
class Sword : public Weapon {};

// concept의 정의
template <typename T>
concept IsWeapon = std::is_base_of_v<Weapon, T>;

template <typename T> requires IsWeapon<T>
void Foo(T weapon) { ... }

구조적 제약 컨셉

requires 키워드와 함께 사용해 템플릿 타입이 '어떤 구조를 가진 타입'이어야 한다는 구조적 제약을 규정한다. (ex: Foo 라는 멤버를 가진 타입만 올 수 있음)
concept 이름 = requires(가상의 인스턴스) { 유효한지 확인할 표현식 }; 꼴로 작성한다.
bool을 반환하는 방식이 아니라 SFINAE를 통해 표현식의 유효성 여부를 검증하는 방식임에 주의하자.

다음은 구조적 제약 컨셉의 몇 가지 예시들이다.

template <typename T>
concept Addable = requires(T a, T b)
{
	a + b; // 이 식이 성립하는지 체크한다.
};

template <typename T>
concept HasName = requires(T t)
{
	t.name; // name이란 이름의 멤버를 가지고 있으며, 현재 context에서 접근 가능한지 체크한다.
}

template <typename T>
concept HasSubtypeFoo = requires(T t)
{
	typename T::Foo; // Foo라는 타입이 내부에 정의되어 있는지 체크한다.
}

template <typename T, typename U = T>
concept Swappable = requires(T&& t, U&& u)
{
	std::swap(std::forward<T>(t), std::forward<U>(u)); 
	std::swap(std::forward<U>(u), std::forward<T>(t)); // std::swap으로 스왑 가능한지 체크한다.
}

형식 제약과 구조적 제약을 구분하자

자주 발생하는 실수가 구조적 제약인 요구 조건 표현식에 형식 제약을 넣으려고 하는 것이다.
requires 요구 조건 표현식에 의해 구현되는 구조적 제약은 어디까지나 '이 식이 문법적으로 유효한가'만을 평가하지 '이 식이 결과적으로 true를 반환하는가 false를 반환하는가'를 평가하는 게 아니라는 것을 명심해야 한다.

// 예컨대, 다음 요구 조건 표현식은 T가 정수형이 아니어도 std::integral<T>가 false를 반환하므로 문법적으로 오류가 없고, 따라서 모든 T에 대해 통과된다.
template <typename T>
concept Requirements = requires(T t)
{
	std::integral<T>;
}

// 정수형만 통과시키려면 형식 제약 컨셉을 정의하는게 좋다.
template <typename T>
concept Requirements = std::integral<T>;

형식 제약과 구조적 제약을 동시에 걸기

만약 서로 종류가 다른 두 제약을 동시에 걸고 싶다면, 우선 requires 요구 조건 표현식을 쓰고 그 안에 다시 requires 형식 조건 라인을 작성하면 된다.

template <typename T>
concept HasKeyWhichInt = requires(T t)
{
	t.value; // (1) t는 value 멤버를 가지며, 현재 context에서 접근할 수 있는가?
	requires std::same_as<decltype(t.key), int>; // (2) t는 key 멤버를 가지고, 그 멤버는 int 타입인가? t가 key라는 멤버를 가지지 않거나, int가 아니라면 조건에 부합하지 않는다.
}

혹은, 그냥 두 컨셉을 각각 만들어서 두 개를 AND로 묶어도 상관 없다.

// 구조적 제약 컨셉
template <typename T>
concept Addable = requires(T a, T b)
{
	a + b;
};

// 형식 제약 컨셉
template <typename T>
concept IsIntegral = std::integral<T>;

// 두 개를 합친 컨셉
template <typename T>
concept UnionRequirement = Addable<T> && IsIntegral<T>;

requires

제약조건을 제시하는 키워드. requires 뒤에 제약조건을 쓰는 것으로 사용된다.

  1. 템플릿 키워드 뒤에서 사용되면 뒤에 오는 표현식을 제약조건으로 제시한다. bool을 반환하는 상수 표현식, concept, type_traits나 concepts 헤더에 정의된 제네릭 함수 어떤 것이든 오거나 논리 연산자로 조합해 사용할 수 있다.
  2. 위에서 보았듯이 concept을 정의할 때 요구 조건 표현식 내에서 형식 제약을 거는데 사용할 수 있다.
  3. '컴파일 타임 술어(Compile-time Predicate)'도 제약조건으로 제시할 수 있다.

어디까지나 제네릭 프로그래밍이므로 제약 조건에 부합하지 않으면 에러를 내는 것이 아니라 SFINAE 규칙에 따라 해당 템플릿을 후보에서 제외할 뿐이다.
키워드가 오는 위치는 temaplate<> 키워드 뒤에, 또는 함수의 서명 뒤에 올 수 있다.

template <typename T> requires std::is_integral_v<T> // 여기 올 수도
void Foo(T arg)
{ ... }

template <typename T>
void Foo(T arg) requires std::is_integral_v<T> // 여기 올 수도 있다. 이 경우엔 '후행 requires 절'이라고 부른다.
{ ... }

requires 절은 템플릿의 서명에 포함되므로, 선언부와 정의부에 동일하게 적어야 한다. VS가 자동으로 함수 정의부를 만들어줄 때 이걸 빼먹으니 주의하기 바람.

예시

다음은 requires를 활용하는 몇 가지 예시들이다.

template <typename T> requires std::is_integral_v<T>
void Foo(T arg) { ... }

double Foo(double arg) { ... }

Foo(1); // 1번 함수. is_integral_v<int>가 true를 반환하여 템플릿 구체화.
Foo(3.14f); // 2번 함수. is_integral_v<float>이 false를 반환하여 템플릿 구체화가 일어나지 않음. 그 다음 후보인 2번 함수에 인자가 암시적 변환되어 전달됨.

// 템플릿 오버로딩
#include <concepts> // 여러 기정의된 컨셉트용 템플릿 함수 제공

// 1. 임의 접근이 가능한 컨테이너의 이터레이터를 받는 경우 -> 포인터 더하기 연산으로 이동
template <typename T> requires std::random_access_iterator<T>
void MoveForward(T& iter, int amount)
{
	iter = iter + amount;
}

// 2. 임의 접근이 불가능한 컨테이너의 이터레이터를 받는 경우 -> 다음 이터레이터로 반복 이동
template <typename T> requires std::input_iterator<T>
void MoveForward(T& iter, int amount)
{
	while (amount--) ++iter;
}

std::vector<int> v = { 1, 2, 3, 4, 5, 6 };
std::list<int> l = { 1, 2, 3, 4, 5, 6 };
MoveForward(std::begin(v), 3); // 1번 조건 통과 -> 구체화
MoveForward(std::begin(l), 3); // 1번 조건 실패 -> 2번 조건 통과 -> 구체화
Note

같은 접근 순위의 템플릿 오버로딩 간에는 위에서부터 아래로 내려가며 순서대로 구체화를 시도한다는 걸 명심하자.

컴파일 타임 술어

bool을 반환하는 조건식을 직접 작성하는 것. 괄호()로 묶어야 한다.
컴파일 타임 술어는 일반적인 표현식과 형태가 같지만, constexpr 함수처럼 컴파일 시점에 컴파일러가 연산을 수행해 그 결과로 치환하는 형태이다.

template <int N> requires (N >= 0)
void Foo() { ... }

템플릿 매개변수에 컨셉 지정하기

requires 키워드를 사용하지 않고 템플릿 매개변수 자리에 바로 컨셉을 넣는 축약 형태도 허용된다.

// 1. 일반적인 템플릿에 사용
// typename이나 class 대신 컨셉을 마치 타입을 적는 것처럼 넣는다.
template <std::integral T>
void Foo(T arg) { ... }

// 2. auto 단축 템플릿에도 사용 가능
// 이 경우 다음과 같이 '컨셉 + auto' 덩어리를 타입처럼 취급하면 된다.
std::integral auto Foo(std::integral auto arg) { ... }

// 3. 템플릿 특수화에도 사용 가능
template <typename T> struct Foo { ... };
template <std::integral T> struct Foo<T> { ... }; // 정수 타입에 대한 특수화

// 4. 가변 인수 템플릿에도 사용 가능
template <std::integral... Args>
void Foo(Args... args) { ... }



// 이를 requires 절을 사용하는 기본 형태로 작성하면 다음과 같다.
// 1. 일반적인 템플릿
template <typename T> requires std::integral<T>
void Foo(T arg) { ... }

// 2. auto 단축 템플릿
std::integral auto Foo(auto arg) requires std::integral<decltype(value)> { ... }

// 3. 템플릿 특수화
template <typename T> struct Foo { ... };
template <typename T> requires std::integral<T> struct Foo<T> { ... };

// 4. 가변 인수 템플릿에도 사용 가능
// 가변 인수 템플릿을 requires로 제약을 걸려면 컴파일 타임 술어와 템플릿 파라미터 팩을 사용해 모든 인자에 대해 체크해야 한다.
template <typename... Args> requires (std::integral<Args> && ...)
void Foo(Args... args) { ... }

C++이 제공하는 다양한 컨셉

다음은 <concepts> 에서 제공하는 컨셉의 목록이다. 대부분 type_traits를 사용한 메타프로그래밍 기법으로 구현되어 있다. 다만 훨씬 편하게 정리되어 있으므로 type_traits 대신 concepts를 사용할 것을 권장한다.

std::integral<T> & std::floating_point

T가 정수 타입인지, 부동소수점 타입인지 확인한다.

std::signed_integral<T> & std::unsigned_integral<T>

T가 부호 있는 정수 타입인지, 부호 없는 정수 타입인지 확인한다.

std::same_as<T, U>

두 타입이 서로 같은 타입인지 확인한다. 아래와 같이 구현되어 있다. 자리가 바뀐 경우도 확실히 해놨음.

namespace detail {
	template <class T, class U>
	concept SameHelper = std::is_same_v<T, U>;
}

template <class T, class U>
concept same_as = detail::SameHelper<T, U> && detail::SameHelper<U, T>;

std::derived_from<D, B>

D가 B를 상속했는지, 업캐스팅이 가능한지 확인한다. 다음과 같이 구현되어 있다.

template <class Derived, class Base>
concept derived_from =
  std::is_base_of_v<Base, Derived> &&
  std::is_convertible_v<const volatile Derived*, const volatile Base*>;
