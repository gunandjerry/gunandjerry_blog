(0) 사소한 문제 해결 사례 - 엔진, 에디터, 빌드

프로젝트 강제 재빌드가 필요할 때

희안한 실수를 하거나 엔진 버전이 업데이트되는 등 기존 프로젝트의 엔진 빌드를 제거하고 다시 빌드해야 하는 경우가 있다. 이 때는 Intermediate(빌드된 엔진 파일들이 들어있음), Saved(보통은 상관 없지만, 내부에 빌드 관련 정보를 저장하는 경우가 있음), Binaries, .vs, .sln 폴더와 파일들을 제거하고 uproject 파일을 통해 재빌드를 하면 된다.

특정 엔진 버전이 인식되지 않을 때

런처경로\Engine\Binaries\Win64\UnrealVersionSelector.exe 복사해서
등록할 엔진경로\Engine\Binaries\Win64에 붙여넣고 실행 -> Success to regist 뜨면 잘 등록된 것.

Error Expected XXX.h to be first header included

언리얼 빌드 툴 규약에 따라 cpp 파일에는 자기 헤더가 가장 위에서 포함되어야 한다.

VS를 통해 빌드할 때 ‘이 환경에선 Client를 빌드할 수 없다’는 에러가 뜨는 경우

Build Configuration을 에디터로 지정하지 않은 것. 기본 VS IDE 세팅에선 Config창이 좁아서 글자가 다 안 보인다.

상단바 우클릭→ Customize → Commands → Toolbar → Standard → Solution Configurations - Modify Selection - Width에서 넓힐 수 있다.

에디터에서 Nav Mesh 베이킹이 안 되는 문제

Dynamic Nav Mesh 옵션을 켜고, Navigation Invoker 컴포넌트가 붙은 액터를 레벨에 배치했는데 Nav Mesh가 동적으로 베이킹이 안 됐다. → Nav Mesh 베이킹이 워낙 무거운 동작이라 그런지 관련 버그가 참 많다. 이 경우엔 컴포넌트를 제거했다가 다시 다니까 작동했다.

아니면, 실제론 베이킹이 됐는데 시각적으로 반영만 안 된 것일 수도 있다.

델리게이트에 함수 바인딩할 때 가려진 부모 함수와 이름이 같은 함수를 등록하면 부모 함수를 호출하는 문제

말 그대로다. 왜 이렇게 되는 지는 모르겠다. 아마 내부적으로 객체의 부모부터 내려오면서 같은 이름을 가진 함수를 찾나보다.

콜라이더가 없는 스태틱 메쉬에 소켓을 추가하면 메쉬가 증발하는 문제

고의인지 버그인 지는 모르겠으나, 콜라이더를 추가해야만 한다.

디테일 탭의 슬롯에 에셋 바인딩이 안 되는 문제

아마 에셋 경로에 한글이나 괄호같은 특수문자가 포함되어 있을 것이다. 그럼 안 된다.

DYNAMIC_DELEGATE만 따로 독자적인 헤더로 빼고 사용할 수 있을까

안 된다. 다이나믹 델리게이트의 경우 블루프린트에서도 바인딩할 수 있어야 하므로 UHT가 파싱할 수 있어야 하며, 따라서 generated.h 를 만들어내는 특정 클래스와 함께 선언해야 한다.

UPhysicalMaterial 타입을 사용했는데 링크 오류가 발생

UPhysicalMaterial같은 언리얼 물리 에셋을 사용하려면 PhysicsCore 모듈을 포함해야 한다.

블루프린트에서 Variable Set을 시도할 때 Variable ### is read-only within this context and cannot be set to a new value 에러가 뜨는 경우

아마 함수 내부를 편집하는 중일 것이고, 이 함수가 const라서 클래스의 멤버 변수(즉, 함수의 Local variable이 아닌 변수)를 편집할 수 없기 때문. Local variable로 만들어야 한다.

OnRep 함수가 정의되지 않았다고 하는 문제

OnRep 함수는 UFUNCTION() 지정해야 한다.

Alt 드래그로 복사한 액터의 로직이 작동하지 않는 문제

상호작용으로 열 수 있는 상자 액터를 컨텐츠 브라우저에서 꺼내서 놓으면 잘 작동하는데, 이미 레벨에 배치한 액터를 Alt 드래그로 복사하면 작동을 안 했다.

여러 실험을 해본 결과 결론적으로 상호작용 이벤트 델리게이트에 함수 바인딩하는 동작을 CDO에서 하고 있었는데, 이게 Alt 드래그로 복사한 액터에 대해선 동작을 안 했다.

근데 이 코드를 BeginPlay로 옮기니까 이번엔 Alt 드래그로 복사한 액터만 되고, 컨텐츠 브라우저에서 꺼낸 액터는 안 되는 해괴망측한 경우가 생겼다.

AddDynamic 시점의 atomic 크래쉬

Exception thrown: read access violation. **std::_Atomic_address_as<__int64,std::_Atomic_padded<unsigned __int64> >**(...) returned 0x6F8

위와 같은 에러가 Dynamic Multicast Delegate에 AddDynamic을 호출할 때 발생.
AddDynamic을 시도한 델리게이트가 nullptr일 때 발생한다.

Media Sound Player가 Level Travel 후에도 재생되고 있는 문제

Media Sound Player 컴포넌트에 Auto Activate가 켜져있으면 디폴트 슬롯에 부착된 오디오가 자동으로 재생되는 게 문제였다.

타이틀 폰이 어느 순간부터 갑자기 생성되지 않는 문제

아래와 같이 플레이어 폰을 생성하는 가상함수를 BP에서 오버라이딩해놓고 부모 함수를 호출하지 않고 있었다.

상당히 주의해야 하는 부분인데, AActor::BeginPlay 같이 따로 부모 가상함수를 호출하지 않아도 되는 이벤트 노드가 있는가 하면 위와 같이 오버라이딩을 해버리기 때문에 부모 함수를 명시적으로 호출해줘야 하는 노드도 있다.

이 차이는 두 경우의 처리 방식이 다르기 때문이다. 전자는 부모의 함수에서 마지막에 'BlueprintImplementableEvent'를 따로 호출해주는 방식이고, 후자는 생으로 virtual 함수를 BP에서 오버라이드할 수 있게 해둔 것이다.

생성된 폰의 카메라가 {0, 0, 0} 방향을 바라보는 문제

카메라의 회전을 담당해야 하는 SpringArmComponent의 Inherit Pitch, Yaw, Roll 설정을 꺼놨기 때문이다. 컨트롤러가 바라보는 방향(PlayerStart 방향으로 초기화되는)을 상속하지 못 하니 {0, 0, 0}을 바라보고 있을 수밖에.

체스트 액터를 열면 더이상 FHInteractComponent의 ComponentTick이 호출되지 않는 문제

상자를 열면 더는 상호작용이 안 되게 하겠다고 InteractComp->SetActive(false);를 쓴 게 문제였다.
모든 ActorComponent는 bIsActive 플래그를 가지며, 이게 꺼져있으면 컴포넌트가 비활성 상태라는 뜻이니 틱이고 뭐고 아무것도 동작하지 않는다.
실제로 컴포넌트들을 부착해보면 시작과 동시에 활성화가 되어야 하는 애들만 Auto Activate 체크가 되어있는 걸 볼 수 있다.

래그돌로 전환이 안 되는 문제

스켈레탈 메쉬의 Collision이 NoCollision일 땐 전환이 안 된다.
